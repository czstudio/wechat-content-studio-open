# WeChat Content Studio Open

把微信公众号写作，从“碰运气写一篇”，变成“有流程、有风格、有审稿、有返工、有发布闭环”的内容生产系统。

如果你是下面这几类人，这个 skill 会非常适合你：

- 想做公众号，但每次都卡在“写什么”
- 能写一点，但总觉得文章像 AI、像汇报、像流水账
- 有一堆素材、笔记、截图、聊天记录，却不会组织成文章
- 想模仿一种文风，但不想把 skill 锁死在某个作者身上
- 想要的不只是“帮我生成文章”，而是“帮我把整条内容链做出来”

## 1 分钟快速上手

### 情况 A：你完全没思路

你说：

```text
帮我做 3 个公众号选题，主题是 ReAct 框架局限性，今天最好能开写。
```

它会给你：

- 2-4 张 `Topic Cards`
- 哪个题最值得先写
- 哪个题先别碰
- 哪个题虽然有意思，但素材还不够

### 情况 B：你已经有方向，想直接开写

你说：

```text
这个方向定了，直接给我开写 brief。
```

它会给你：

- `Draft Brief`
- 开头该怎么起
- 三段骨架怎么走
- 哪些素材必须上
- 哪些地方不能写偏

### 情况 C：你已经有初稿，只想发前 review

你说：

```text
这篇已经有初稿了，帮我发前 review，不要重写整篇。
```

它会给你：

- `Proof Gate`
- `Support Matrix`
- `Repair Brief`

也就是：

不只告诉你“哪里不好”，还告诉你“优先改哪三刀”。

## 这不是普通写作 prompt，它是一个公众号生产操作系统

普通 AI 写稿工具通常只做到这里：

- 给你几个标题
- 直接生成一篇稿
- 顺便模仿一下语气

然后问题就来了：

- 选题不准
- 开头很空
- 中段没证据
- 结尾没落点
- 看起来很完整，其实不能发

这个 skill 解决的是整条链：

`Topic Board -> Draft Brief -> Draft -> Proof Gate -> Repair Brief -> Image Plan -> Publish Prep`

一句话理解：

它不只是“帮你写”，而是“帮你从想法走到可发草稿”。

## Before / After

| 以前的 AI 写稿 | 现在这套系统 |
|---|---|
| 给你几个标题，然后直接开写 | 先做 `Topic Board`，先判断值不值得写 |
| 能生成初稿，但结构容易飘 | 先给 `Draft Brief`，写作有骨架 |
| 文章像 AI，但说不出为什么 | `Proof Gate` 直接指出 AI 味、支撑链、标题承诺问题 |
| review 之后还是不会改 | `Repair Brief` 告诉你优先改哪几刀 |
| 写完停在 Markdown | 后面还能接 `Image Plan` 和 `Publish Prep` |
| 只能模仿固定作者 | 你可以自己带样文、seed、素材库，训练自己的 DNA |

一句话：

以前是“一次性生成”。

现在是“可复用生产链”。

## 小白最能感受到的 7 个爽点

### 1. 不知道写什么？先给你“可比较的选题卡”

不是只丢 5 个标题给你。

它会给你 2-4 个真正能拿来开会讨论的 `Topic Cards`，每张卡都带：

- 这篇是写给谁的
- 读者现在卡在哪
- 为什么这题会打中人
- 开头第一屏怎么起
- 文章骨架怎么走
- 现在有没有足够素材
- 值不值得今天就开写

也就是说，你第一次就能区分：

- 哪个题“看着高级但写不出来”
- 哪个题“今天就能写”
- 哪个题“得补素材，不然会空”

### 2. 不知道怎么模仿文风？你自己带样文就行

这个 skill 没有把风格锁死在任何作者身上。

你可以自己提供：

- 3-10 篇你自己的历史文章
- 1-3 篇你想模仿的样文
- 一份 style seed
- 一份品牌 / 社群语气说明

它会把这些合起来，抽成你的写作 DNA。

不是“像某个固定博主”，而是“像你想要的那个版本的你”。

### 3. 不是直接乱写，而是先给你 `Draft Brief`

很多 AI 一上来就写整篇，结果：

- 方向对，结构不稳
- 开头有点意思，但越写越散
- 看似很满，实则没有主线

这里先用 `Draft Brief` 卡住写作：

- 这篇到底承诺读者什么
- 开头用哪个场景
- 三段骨架怎么走
- 哪些素材必须用
- 哪些方向绝对不能跑偏

这一步对小白非常有用，因为它等于帮你把“写作恐惧”拆成了可执行动作。

### 4. 有初稿也不怕，它会帮你做“真 review”

不是那种“挺好的，可以更自然一点”的假 review。

它会直接检查：

- 有没有 AI 味
- 开头能不能把人拉住
- 中段有没有真实支撑
- 结尾是不是飘着
- 标题和正文是不是对得上
- 这篇到底能不能发

最关键的是，它不是只有意见，它还会继续给你：

- `Support Matrix`
- `Repair Brief`

也就是：

它不只是告诉你“哪不行”，还告诉你“先改哪三刀最值”。

### 5. 你有一堆素材，它会帮你“捞出能直接入稿的东西”

很多人不是没内容，而是内容都散在这里：

- 笔记
- 对话
- 项目记录
- 采访
- 截图
- 旧文章

这个 skill 有素材召回思路：

- 帮你找适合开头的片段
- 找适合中段证明的片段
- 找适合结尾升华的片段

甚至会输出接近可直接上卡片的“素材包”。

这对内容型创作者特别重要，因为它把“我好像有东西可写”变成“我知道哪段能用”。

### 6. 它连返工都替你想好了

很多工具在 review 后就结束了。

但现实里最痛苦的，不是写，而是二修三修：

- 哪里要重写？
- 是全篇推翻还是局部补强？
- 是缺证据还是缺节奏？
- 是选题错了还是只是结尾没收住？

这里专门有 `Repair Brief`：

- 主故障是什么
- opening / middle / ending 哪块塌了
- 哪 1-3 个问题必须先修
- 具体改哪段

所以它是“可返工系统”，不是“生成完就不管”。

### 7. 最后还能接配图、排版、发布

很多写作工具最后都停在 Markdown。

但公众号真正要发出去，还差很多：

- 封面
- 插图
- 信息图
- 排版
- HTML
- 草稿箱

这套 skill 的目标不是停在“写完”，而是能继续往下接：

- `Image Plan`
- `Publish Prep`

你可以自己换图像后端、自己换发布后端，但这个流程骨架已经给你搭好了。

## 它到底包含哪些功能

| 功能 | 解决什么问题 | 为什么小白会觉得值 |
|---|---|---|
| Topic Board | 不知道写什么 | 不再只看标题，能看清“这个题能不能写” |
| DNA Calibration | 不知道怎么写出自己的味道 | 用样文和 seed，而不是凭感觉 |
| Draft Brief | 不知道怎么开写 | 先把写作变成执行单 |
| Draft | 出初稿 | 按 DNA 和 brief 写，不乱飞 |
| Proof Gate | 不知道文章能不能发 | 发前有硬质检 |
| Repair Brief | review 后不会改 | 直接给返工路径 |
| Material Recall | 明明有素材但不会用 | 把散素材变成可用证据 |
| Image Plan | 文章配图乱 | 明确封面 / 插图 / 信息图怎么分工 |
| Publish Prep | 写完发不出去 | 给排版和发布交接件 |

## 这个 skill 吸收了哪些强能力

它不是从零拍脑袋写的，而是融合了几类很强的能力：

| 来源 | 吸收的核心 |
|---|---|
| Khazix 风格写作 | 更像活人、更会开头、更会去汇报味 |
| Huashu 式审稿 | 多轮 review、checklist、可复用审稿流程 |
| Baoyu 体系 | 封面、插图、信息图、发布闭环 |
| 原 WeChat 写作栈 | DNA-first、topic -> outline -> draft -> title 的骨架 |

所以它强的地方不是“某一个 prompt 很巧”，而是：

每个环节都吸了成熟能力，然后重新编排成了一整套工作流。

## 适合哪些人

### 个人作者

- 想稳定更新公众号
- 想逐渐形成个人文风
- 不想每次都从空白开始

### 社群 / 品牌内容负责人

- 想让内容更稳定
- 想把品牌语气沉淀成可复用资产
- 想降低“换一个写手就全变味”的风险

### 代写 / 编辑 / 内容团队

- 需要 topic handoff
- 需要 draft handoff
- 需要 review handoff
- 需要能把意见落成动作

### AI 写作重度用户

- 已经不满足于“一键生成”
- 需要更强的控制力、质检能力、返工能力

## 你甚至可以把它当成“内容副驾驶”

这个项目不是要求你一上来就搭完全部流程。

你完全可以先只拿它做一件事：

- 只拿它做选题
- 只拿它做 review
- 只拿它做返工

所以它既适合：

- 小白拿来先解决一个痛点
- 也适合团队拿来做完整流程

## 它和“模仿某个作者”的 skill 最大区别

很多 skill 的本质是：

“请像某人一样帮我写。”

这个 skill 的本质是：

“请帮我构建一个我自己的公众号生产系统。”

这两个层级差很多。

前者更像一次性 cosplay。
后者更像长期可复用的内容基础设施。

## 为什么这个项目会有传播性

因为它不是只解决“写作”。

它一次性解决了内容生产里最痛的几个问题：

- 不知道写什么
- 不知道怎么写出自己的风格
- 不知道文章到底能不能发
- 不知道 review 以后怎么改
- 不知道怎么把文章真正做成产品

也就是说：

它既有“功能”，也有“噱头”：

- 功能上，它很全
- 传播上，它是一个很容易被理解的概念：
  - “把公众号写作做成操作系统”
